# Dependency Inversion Principle in wage-comparator

> High-level modules (components, pages, feature orchestration) should depend on abstractions, not on concrete low-level implementations (Supabase client, fetch calls, third-party SDKs). The abstractions in a React + TypeScript codebase are hooks and interfaces — not DI containers, not factories, not service locators. This file explains DIP in the form it actually takes in this project.

## Why DIP looks different in React

Most SOLID material describes DIP in terms patterns React does not use: constructor injection, dependency containers, `@Inject` decorators, service registries. None of that applies here. In a hooks-based React codebase, DIP is achieved by ensuring that high-level units (pages, feature components) depend on *stable hook contracts* rather than on the concrete clients those hooks wrap.

The principle still holds — components do not import `supabaseClient` directly, they consume `useGetWagesQuery` — but the *vocabulary* of inversion is different. The "interface" being depended on is the hook signature; the "implementation" is the RTK Query endpoint that owns the actual Supabase call. Swapping Supabase for another backend means rewriting the endpoint file; the components keep working without modification.

## DIP in this project, by example

### Data access: `architecture.md` §4 already encodes the inversion

`architecture.md` §4 lists data flow as a strict chain:

```
Component → RTK Query hook → endpoint → Supabase
```

The arrows go one direction. Components depend on hooks; hooks are generated by RTK Query endpoints; endpoints know about Supabase. This is exactly the dependency inversion: the high-level unit (the component) depends on the abstract surface (the hook), not on the concrete client (Supabase).

The ADR in `architecture.md` §5 reinforces this by forbidding `useEffect` for data fetching and forbidding raw Supabase calls in components or hooks. Both prohibitions are DIP enforcement: a `useEffect` with a direct Supabase call would make the component depend on the concrete client, bypassing the inversion the endpoint layer was put in place to provide.

The practical consequence: if Supabase is ever replaced (with a custom REST API, with a different BaaS, with a self-hosted Postgres + REST), the change ripples through `features/*/api/*.ts` files only. Pages, components, and hooks downstream of the endpoint do not need to change as long as the endpoint's input/output contract is preserved.

### Authentication: components consume `useAuth`, not the auth client

`architecture.md` §2 documents `AuthContext` / `AuthProvider` / `useAuth` as the three-file split for the auth context. Components that need session info call `useAuth()`. They do not import `supabase.auth` directly.

The inversion is intentional: `useAuth` is a stable contract (returns `{ user, signIn, signOut, isLoading }` or similar). The underlying implementation can use Supabase auth today, magic-link tomorrow, OAuth providers later — `useAuth`'s shape stays the same, and every component depending on it stays unchanged.

### Premium gating: another instance of the same pattern

The OCP file already shows `useFeatureAccess` as the single point of change for new tiers. Read through the DIP lens: components depend on the abstraction (`useFeatureAccess()` returns capability flags), not on `user.app_metadata.tier === 'premium'` inline. The implementation detail (where the tier lives, how it is computed) is owned by the hook, not by every consumer.

## A pattern to watch for: hooks that leak their backend

A subtle DIP violation: a hook is built, but its return shape leaks the underlying backend's concepts:

```tsx
// Smell — useWageRecords returns Supabase's response shape directly
export function useWageRecords(country: string) {
  return useGetWagesQuery({ country });
  // RTK Query's auto-generated return is { data, isLoading, error, refetch, ... }
  // — but the `data` shape may be Supabase's row shape, with column names like
  // `monthly_wage_eur` and `gender_str` that betray the database schema.
}
```

Components downstream now depend on Supabase's column names. Renaming a column in the database breaks every component reading it. The DIP fix lives in the endpoint: transform the response into a domain-shaped type (`WageRecord` with `monthlyWage`, `gender`) before exposing it.

```tsx
// features/salary-calculator/api/wageApi.ts
getWages: builder.query<WageRecord[], { country: string }>({
  query: ({ country }) => `/wages?country=eq.${country}`,
  transformResponse: (raw: SupabaseWageRow[]) =>
    raw.map((r) => ({
      monthlyWage: r.monthly_wage_eur,
      gender: r.gender_str,
      // ...
    })),
}),
```

Now the contract that components depend on is `WageRecord` (a domain type), not `SupabaseWageRow` (an infrastructure detail). The inversion holds.

## A pattern to watch for: leaking the backend via type imports

The same leak can happen through TypeScript imports without any runtime code:

```tsx
// Smell — component imports the Supabase row type
import type { Database } from '@/shared/lib/supabase-types';
type WageRow = Database['public']['Tables']['wages']['Row'];

export function WageCard({ wage }: { wage: WageRow }) { ... }
```

The component is now coupled to the Supabase types file at the type level. If the table is ever renamed or restructured, the component changes. DIP-aligned: domain types live in `shared/types/` or in the feature, and the endpoint translates infrastructure types to domain types. Components only ever see domain types.

## Where DIP does NOT apply

This is the principle most prone to over-application. Not every import is a "dependency" worth inverting:

- A component importing `cn()` from `shared/lib/cn` is not violating DIP. `cn` is a pure function with no policy and no replaceable implementation. It is a utility, not a service.
- A component importing `Button` from `shared/components/ui/` is not violating DIP. `Button` is a shared primitive, not a backend or an external service. Components depend on it because it *is* the abstraction they need.
- A hook importing `useDispatch` from `react-redux` is not violating DIP. The Redux store is the abstraction; bypassing `useDispatch` would mean reaching into the store directly, which is the actual violation.

The rule of thumb: DIP applies to dependencies that *could realistically change* and where the high-level unit *should not have to change with them*. Utilities, design-system primitives, and framework hooks do not meet that bar. External services, backends, and concrete clients do.

## Testability is a consequence, not the goal

A common mistake in DIP-for-React material is framing the principle as "depend on abstractions so you can mock them in tests." That framing is wrong because it inverts cause and effect.

DIP is about **contract stability**: high-level code should not break when low-level implementation changes. Testability falls out naturally — code that depends on a stable hook contract is easy to test because you can mock the hook — but the *reason* to apply DIP is not "to enable tests." If a hook's contract is unstable, no amount of mocking saves the components that depend on it; they still break when the contract changes.

When deciding whether a hook needs to be inverted further, ask: *can this hook's implementation change without breaking the components that depend on it?* If yes, the inversion is in place. If no, the contract is too coupled to the implementation, and that is the actual problem — testability concerns are a symptom of it, not the disease.

## When NOT to invert

DIP, like every SOLID principle, can be applied prematurely:

- A hook used by exactly one component, wrapping a single concrete call, is not a DIP candidate. The "abstraction" is theatrical — there is only one implementation and one caller.
- A type imported across two feature files might not warrant an abstract intermediate. If both features genuinely need the same shape and no third backend is on the horizon, the direct dependency is fine.
- A utility function shared by many callers does not need to be abstracted further. It is already the abstraction.

`architecture.md` §3 applies again: prefer the smaller, more reversible choice. Inversion that does not buy real protection against change is just indirection that costs reading effort.

## What this file does NOT do

- It does not introduce dependency injection containers, service locators, or factory patterns. None of those are idiomatic React, and this codebase does not use them.
- It does not require every external dependency to be wrapped in a custom hook. RTK Query's auto-generated hooks already provide the inversion for data access; wrapping them further would be redundant.
- It does not contradict the existing `useAuth` / `useTheme` / `useFeatureAccess` shapes — those are already DIP-compliant abstractions.
- It does not enforce DIP as a testability tool. Testability is a side effect of good contract design, not the reason to invert.

If those rules ever appear in this file in a future edit, that edit is wrong and should be reverted.
